第一章 · 项目定位

一句话定义

FunASR 是阿里巴巴达摩院开源的端到端语音识别（ASR）工具包，由 ModelScope 社区维护，核心目标是让开发者用一行代码完成"语音转文字 + 说话人分离 + 情感识别 + 标点添加"的全套流程。

2023 年发表在语音顶会 INTERSPEECH，是目前中文开源 ASR 领域影响力最大的项目之一。

指标数值意味着什么

语言覆盖50+含中/英/日/韩/粤语及方言
模型参数0.4M ~ 1.7B从轻量 VAD 到 LLM 级大模型全覆盖
开源协议MIT商用无限制

2.2 核心模型矩阵详解

（1）ASR（语音识别）模型

模型架构思想参数量语言速度(GPU)独特能力
SenseVoiceSmall非自回归 CTC + 情感头234M中/英/日/韩/粤170x情感识别 + 音频事件检测
Paraformer-Large非自回归 Transformer220M中/英120x高精度中文生产级 ASR
Fun-ASR-NanoLLM-based (SenseVoice + Qwen3-0.6B)800M31 语言17x最高精度，LLM 驱动
Qwen3-ASRQwen3 架构0.6B/1.7B52 语言—最广语言覆盖
GLM-ASR-NanoGLM 架构1.5B17 语言—方言支持
Whisper-large-v3自回归 Encoder-Decoder1550M多语言13x翻译 + ASR
Whisper-large-v3-turbo自回归 Encoder-Decoder809M多语言46x翻译 + ASR（加速版）

关键概念——「非自回归」vs「自回归」：

  • 非自回归（Paraformer / SenseVoice）：一次性并行输出整句文字，速度极快（120-170x 实时），适合生产环境
  • 自回归（Fun-ASR-Nano / Whisper）：逐 token 生成，慢但精度更高，适合对准确率有极致要求的场景

（2）VAD（语音活动检测）

fsmn-vad：仅 0.4M 参数的轻量模型，负责把一段长音频切成有语音的片段。

为什么需要 VAD？——真实录音里大量时间是沉默、背景噪音。VAD 把"有人说话的部分"切出来，只把有效片段送给 ASR 模型，大幅节省算力。

最新版本引入了动态 VAD（Dynamic VAD），自适应静音阈值：短句保持完整不被切断，长段自动按语义拆分。

（3）标点恢复模型

ct-punc：290M 参数，基于 Transformer。ASR 输出的原始文字是连续的（"我们讨论一下三季度计划好的我有三点意见请说我们有半小时"），标点模型自动加回标点（"我们讨论一下三季度计划。好的，我有三点意见。请说，我们有半小时。"）。

（4）说话人分离

cam++：7.2M 参数，基于说话人声纹嵌入。解决"谁在说话"的问题——会议录音里 5 个人轮流发言，cam++ 自动给每句话打上 Speaker 0 / Speaker 1 / … 标签。

（5）情感识别

emotion2vec+large：300M 参数，从语音中识别情感标签（开心/悲伤/愤怒/中性），粒度可配到 utterance 级别（单句话的情感）。

第三章 · 统一调度机制——为什么说"一行代码"

这是 FunASR 最精妙的设计。传统 ASR 流程需要开发者手动串联多个模型：

音频 → VAD 切分 → ASR 识别 → 标点恢复 → 说话人标注 → 结果

FunASR 的 AutoModel 自动完成整条链路：

from funasr import AutoModel

model = AutoModel(
    model="paraformer-zh",      # ASR 模型
    vad_model="fsmn-vad",       # 语音活动检测
    punc_model="ct-punc",       # 标点恢复
    spk_model="cam++",          # 说话人分离
    device="cuda"
)
result = model.generate(input="meeting.wav")

背后发生了什么：

  1. fsmn-vad 扫描音频，找到所有有人声的片段
  2. 每个片段送 paraformer-zh 做语音转文字
  3. ct-punc 给输出文字加标点
  4. cam++ 对每个片段的声纹做聚类，打上 Speaker ID
  5. 最终合并输出带时间戳 + 说话人 + 标点的结构化文本

输出示例

[00:00.4 → 00:03.8]  Speaker 0: 我们讨论一下三季度计划。
[00:04.2 → 00:07.1]  Speaker 1: 好的，我有三点意见。
[00:07.5 → 00:12.3]  Speaker 0: 请说，我们有半小时。

OpenAI 兼容 API 的价值

这是一个关键设计选择：API 格式完全兼容 OpenAI 的 /v1/audio/transcriptions 端点。

  • LangChain / Dify / AutoGen / Coze 等 Agent 框架可以零改动接入
  • 任何现有使用 OpenAI Whisper API 的应用，只需把 URL 改成 localhost:8000 即可切换
  • Postman / curl / JavaScript SDK 等现成工具直接可用
# 启动服务
funasr-server --device cuda

# 调用（和 OpenAI 完全一致的格式）
curl http://localhost:8000/v1/audio/transcriptions \
  -F file=@sample.wav \
  -F model=sensevoice \
  -F response_format=verbose_json

8.1 workflows/update-api-docs.yml — 唯一 CI 工作流

技术栈选型
组件选型说明
触发方式push 到 main 分支仅在核心文件变更时触发
路径过滤funasr/**/*.py, gen_api_docs.py, training.html, 自身精准过滤，避免无意义构建
运行环境ubuntu-latestGitHub Actions 标准 Linux 虚拟环境
Checkoutactions/checkout@v4 + fetch-depth: 0拉取完整 Git 历史
Pythonactions/setup-python@v5, 3.11固定版本保证可重复性
部署peaceiris/actions-gh-pages@v3增量推送到 gh-pages 分支
行为keep_files: true增量更新，不删除已有文件
name: Update API Documentation
on:
  push:
    branches: [main]
    paths:
      - 'funasr/**/*.py'
      - 'scripts/gen_api_docs.py'
      - 'training.html'
      - '.github/workflows/update-api-docs.yml'
jobs:
  build-api-docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Generate API docs
        run: python3 scripts/gen_api_docs.py

      - name: Deploy to gh-pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./gh-pages-output
          publish_branch: gh-pages
          keep_files: true
代码实现的功能
name: Update API Documentation给工作流起名，显示在 GitHub Actions 面板和 Badge 上
branches: [main]仅限 main 分支。避免开发中的半成品代码污染线上文档
paths:路径过滤器。只有这些路径下的文件变更时才触发
fetch-depth: 0拉取完整 Git 历史。gh-pages 分支与 main 是独立历史线，peaceiris 需要完整历史才能正确计算增量提交
python-version: '3.11'固定 Python 3.11。避免 ubuntu-latest 预装版本漂移导致文档生成行为不一致
keep_files: true增量更新模式。只覆盖同名文件，保留手动上传的静态资源

路径过滤器的设计意图

路径为什么监听它
funasr/**/*.py核心源码。任何 Python 代码改动可能影响 API 签名
scripts/gen_api_docs.py文档生成脚本本身。脚本改了，生成的文档结构可能也会变
training.html训练文档模板
.github/workflows/update-api-docs.yml工作流自身。工作流逻辑改了需要立即生效
关键洞察：这是整个项目唯一的自动化 CI。它不跑测试、不跑 lint、不做构建——只做一件事：自动生成并部署 API 文档。FunASR 的测试需要 GPU + 大模型，不适合 GitHub Actions 免费配额。团队选择将 CI 资源集中在最有社区价值的交付物上——文档的实时更新。

完整执行链路

开发者 push 代码到 main 分支 ↓ GitHub 检查：变更的文件是否匹配 paths 过滤条件？ ↓ 匹配 拉取完整仓库代码（fetch-depth: 0） ↓ 安装 Python 3.11 ↓ 运行 gen_api_docs.py，产出文档到 ./gh-pages-output/ ↓ peaceiris/actions-gh-pages 将产出目录推送到 gh-pages 分支 ↓ GitHub Pages 自动更新网站内容

Docs or links checked检查改动的文档链接死链、格式错乱
Runtime/deployment command tested实际运行部署命令API 改了但文档里的命令没更新

compileall 而非跑完整测试的原因：FunASR 的测试需要 GPU + 大模型文件。但 compileall 只需要 Python 标准库，零依赖，秒级完成，能拦截 80% 的低级错误（拼写错误、import 路径、缩进问题）。

┌──────────────────────┐ │ config.yaml │ │ （模板选择器中枢） │ └──────┬───────────────┘ │ 路由到 7 个 Issue 模板 ┌────────────────────┼────────────────────────────┐ │ │ │ ask_questions bug_report deployment_help （使用问题） （Bug 报告） （部署求助） │ │ │ error_docs feature_request migration_benchmark （文档修正） （功能请求） （迁移对比） │ │ showcase PULL_REQUEST_TEMPLATE （应用案例） （代码贡献入口） │ update-api-docs.yml （自动文档部署）

四条核心设计原则

入口控制

config.yaml 禁用空白 Issue + 三个外部链接分流基础问题。从源头保证 Issue 区的信噪比。

领域特化

每个模板收集的字段精准匹配问题类型：音频属性给 Bug、端口给部署、对比数据给迁移。

贡献梯度

文档修正（最低门槛）→ 功能建议/案例展示（中）→ 代码 PR（高）。覆盖不同能力的贡献者。

自动化闭环

PR 合并后 update-api-docs.yml 自动更新文档，用户永远看到最新版。
